# Parametric CSF Geometry Guide  
## Variable rounded-rectangle / circle section with optional twist, two internal rebar rows, and paired weight laws

This guide explains how to use the Python generator `writegeometry_rio_v2.py` through its launcher script (`un_writegeometry_v6_twist.sh` on Linux and the corresponding `.ps1` file on Windows).

The intended workflow is simple:

1. Open the launcher script.
2. Change the parameter values at the top of the file.
3. Run the script.
4. Get a CSF YAML geometry file as output.

You do **not** need to edit the Python generator for normal use.

---

## 1. What this generator creates

The generator builds a CSF YAML file with:

- one section at the base: `S0`
- one section at the head: `S1`
- an optional twist of the head section relative to the base
- one hollow main shape per section
- up to **two internal rows of bars**
- optional `weight_laws` appended to the YAML

The main section can vary continuously between:

- a **rectangle** (`R = 0`)
- a **rounded rectangle** (`0 < R < min(dx, dy)/2`)
- an approximate **circle** (`dx = dy` and `R = dx/2`)

This applies independently to both `S0` and `S1`.

---

## 2. What the launcher script does

The Bash launcher you provided defines all input values as variables, then passes them to:

```bash
../../src/csf/utils/writegeometry_rio_v2.py
```

So the launcher is only a parameter block plus the final Python call.

A typical workflow is:

```bash
./un_writegeometry_v6_twist.sh
```

and the script writes:

```text
twist_tower.yaml
```

or whatever filename you set in:

```bash
out="..."
```

On Windows, the `.ps1` launcher follows the same idea: edit the variables in the script, then run it. The parameter meanings are the same.

---

## 3. Geometry concept used by the generator

For each station (`S0` and `S1`), the section is defined by:

- center: `x`, `y`
- overall size: `dx`, `dy`
- corner radius: `R`
- geometric wall thickness: `tg`

From these values, the script builds:

- an **outer contour**
- an **inner contour**, obtained by geometric inward offset

The inner contour is computed as:

- `inner_dx = dx - 2*tg`
- `inner_dy = dy - 2*tg`
- `inner_R = max(R - tg, 0.0)`

So the actual hollow wall thickness is controlled by `tg`.

Important consequence:

- `tg` changes the geometry
- `t_cell` does **not** change the geometry generated by this script

`t_cell` is only used in the polygon name tag, for example:

```text
cell_base@cell@t=0.25
```

or

```text
cell_head_outer@t=0.25
```

This matters for later CSF workflows, but not for the coordinates generated here.

## 4. Parameters you usually change

## 4.1 Longitudinal coordinates

```bash
z0="0.0"
z1="50.0"
```

These are the longitudinal positions of the two stations.

Rules:

- `z1` must be strictly greater than `z0`

---

## 4.2 Base section `S0`

```bash
s0_x="0.0"
s0_y="0.0"
s0_dx="10.5"
s0_dy="10.5"
s0_R="4"
s0_tg="0.25"
s0_t_cell="0.0"
```

Meaning:

- `s0_x`, `s0_y`: center of the base section
- `s0_dx`, `s0_dy`: outer dimensions
- `s0_R`: corner radius
- `s0_tg`: geometric wall thickness used to build the inner contour
- `s0_t_cell`: optional `@t=` tag in the exported polygon name

---

## 4.3 Head section `S1`

```bash
s1_x="0.0"
s1_y="0.0"
s1_dx="5"
s1_dy="7"
s1_R="1.2"
s1_tg="0.25"
s1_t_cell="0.0"
```

Meaning is the same as for `S0`, but for the head section.

This allows the section to taper or change shape from base to head.

---

## 4.4 Twist and discretization

```bash
twist_deg="30"
N="128"
singlepolygon="true"
```

### `twist_deg`
This rotates the **head package only** (`S1`) around its own center.

The script rotates consistently:

- outer contour
- inner contour
- row 1 bar centers
- row 2 bar centers

The base `S0` is not rotated.

### `N`
Number of sampling points used to discretize the rounded-rectangle contour.

Practical effect:

- larger `N` -> smoother curves and better circle approximation
- larger `N` -> larger YAML file

Typical use:

- `N = 32` or `64` for quick tests
- `N = 128` or more for smoother round geometries

### `singlepolygon`
This changes the export format.

---

## 5. `singlepolygon=true` vs `singlepolygon=false`

This is a key switch.

## 5.1 `singlepolygon=true`

The hollow cell is exported as **one single polygon** per station, using:

- outer loop
- inner loop with reversed traversal

Typical names:

```text
cell_base@cell
cell_head@cell
```

or, if `t_cell != 0`:

```text
cell_base@cell@t=...
cell_head@cell@t=...
```

Use this mode when you want the cell represented as a single CSF polygon entity with `@cell`.

---

## 5.2 `singlepolygon=false`

The hollow cell is exported as **two separate polygons** per station:

- outer polygon with `weight = 1.0`
- inner polygon with `weight = 0.0`

Typical names:

```text
cell_base_outer
cell_base_inner
cell_head_outer
cell_head_inner
```

In this mode:

- both loops are exported CCW
- the inner polygon is a geometric void with zero weight

Use this mode when you want explicit outer/inner polygons in the YAML.

---

## 6. How the main shape changes from rectangle to circle

The control parameter is `R`.

### Rectangle
Use:

```bash
R = 0
```

### Rounded rectangle
Use:

```bash
0 < R < min(dx, dy)/2
```

### Circle-like section
Use:

```bash
dx = dy
R = dx/2
```

with a sufficiently large `N`.

Example:

```bash
s0_dx="10.0"
s0_dy="10.0"
s0_R="5.0"
N="128"
```

This produces a polygonal approximation of a circle.

---

## 7. Valid geometric ranges to respect

The script assumes a meaningful hollow section.

For stable use, keep:

```text
dx > 0
dy > 0
tg >= 0
dx - 2*tg > 0
dy - 2*tg > 0
```

Also, for normal rounded-rectangle behavior, use:

```text
0 <= R <= min(dx, dy)/2
```

If you move beyond these ranges, the result can become degenerate or invalid for the intended shape.

---

## 8. Internal bars: two rows

The generator can add two rows of bars:

```bash
n_bars_row1="60"
n_bars_row2="60"
area_bar_row1="0.001257"
area_bar_row2="0.001257"
dist_row1_outer="0.06"
dist_row2_inner="0.06"
rebar_weight="5.71"
```

These rows are generated at both `S0` and `S1`.

---

## 8.1 Row 1

Row 1 is built by offsetting **inward from the outer contour**.

Control parameters:

- `n_bars_row1`: number of bars in row 1
- `area_bar_row1`: area of each bar
- `dist_row1_outer`: inward offset distance from the outer contour

Exported names:

```text
rebar_row1_1
rebar_row1_2
...
```

Each bar is exported as a **square with equivalent area**.

The square is generated from:

```python
side = sqrt(area)
```

so `area_bar_row1` controls the square size.

---

## 8.2 Row 2

Row 2 is built by expanding **outward from the inner contour**.

Control parameters:

- `n_bars_row2`: number of bars in row 2
- `area_bar_row2`: area of each bar
- `dist_row2_inner`: outward offset distance from the inner contour

Exported names:

```text
rebar_row2_1
rebar_row2_2
...
```

Again, each bar is exported as an equivalent square.

---

## 8.3 Bar weight

All bars use:

```bash
rebar_weight="5.71"
```

This value is written as the polygon `weight` for each bar.

So:

- bar geometry comes from `area_bar_row1` / `area_bar_row2`
- bar material participation comes from `rebar_weight`

---

## 8.4 How to disable one or both bar rows

To remove a row, set its count to zero:

```bash
n_bars_row1="0"
```

or

```bash
n_bars_row2="0"
```

The script accepts zero bars.

---

## 9. Optional weight laws

The script can append a `weight_laws` block at the end of the YAML.

Available launcher variables:

```bash
bars_row1_law="np.maximum(0.2, 1.0 - 0.8*(z/L))"
bars_row2_law="np.maximum(0.2, 1.0 - 0.8*(z/L))"
s0_law=""
s1_law=""
```

These are optional strings.

If a law string is empty or blank, it is not written.

Important: the generator does **not** evaluate these expressions. It only writes them into the YAML. They will be interpreted later by the CSF side that reads the weight laws.

---

## 9.1 Row laws

A row law is expanded into one explicit law entry per bar pair.

For example, if:

```bash
n_bars_row1="3"
bars_row1_law="1.0 - 0.5*(z/L)"
```

the YAML will contain entries like:

```yaml
weight_laws:
  - "rebar_row1_1,rebar_row1_1: 1.0 - 0.5*(z/L)"
  - "rebar_row1_2,rebar_row1_2: 1.0 - 0.5*(z/L)"
  - "rebar_row1_3,rebar_row1_3: 1.0 - 0.5*(z/L)"
```

Same logic for row 2.

This makes the YAML explicit: each bar pair gets its own line.

---

## 9.2 Shell law for the tower

The tower shell law is controlled through:

```bash
s0_law="..."
s1_law="..."
```

But in this generator, the shell is exported as **one paired law between the S0 shell polygon and the S1 shell polygon**.

That means:

- if both `s0_law` and `s1_law` are empty -> no shell law is written
- if only one is non-empty -> that law is used
- if both are non-empty -> they must be **exactly identical**

Otherwise the script raises an error.

This is stated directly in the Python code.

---

## 9.3 Which polygon receives the shell law

This depends on `singlepolygon`.

### If `singlepolygon=true`
the shell law is written for:

```text
cell_base@cell
cell_head@cell
```

or, with `t_cell != 0`:

```text
cell_base@cell@t=...
cell_head@cell@t=...
```

### If `singlepolygon=false`
the shell law is written only for the **outer shell polygons**:

```text
cell_base_outer
cell_head_outer
```

or with `@t=...` if `t_cell != 0`.

The inner void polygon does **not** receive a shell law.

---

## 10. How to model tower law + two bar-row laws

This is exactly supported by the script.

You can define:

- one law for row 1
- one law for row 2
- one law for the shell/tower

Example:

```bash
bars_row1_law="np.maximum(0.30, 1.0 - 0.6*(z/L))"
bars_row2_law="np.maximum(0.20, 1.0 - 0.8*(z/L))"
s0_law="1.0 - 0.15*np.exp(-((z-10.0)**2)/(2.0*4.0**2))"
s1_law="1.0 - 0.15*np.exp(-((z-10.0)**2)/(2.0*4.0**2))"
```

This produces:

- explicit paired laws for every row 1 bar
- explicit paired laws for every row 2 bar
- one paired law for the tower shell

## 14. Names generated by the script

## Main shell / cell
Depending on mode, the generator writes names such as:

```text
cell_base@cell
cell_head@cell
```

or:

```text
cell_base_outer
cell_base_inner
cell_head_outer
cell_head_inner
```

and, when `t_cell != 0`:

```text
...@t=<value>
```

## Bars
Bars are always named as:

```text
rebar_row1_1
rebar_row1_2
...
rebar_row2_1
rebar_row2_2
...
```

These names are important because the row laws are attached to those exact polygon names.

## 15. Checks already enforced by the script

The Python generator already checks:

- `z1 > z0`
- `n_bars_row1 >= 0`
- `n_bars_row2 >= 0`
- `area_bar_row1 > 0`
- `area_bar_row2 > 0`
- `dist_row1_outer >= 0`
- `dist_row2_inner >= 0`

It also raises an error if:

- both `s0_law` and `s1_law` are provided
- and they are different

## 16. Practical recommendations

### Use `tg` carefully
Because the inner contour is built from `dx - 2*tg` and `dy - 2*tg`, keep enough wall thickness without collapsing the inner void.

### Use `N` according to shape
For nearly circular sections, use a larger `N`.

### Keep laws simple while testing
At first, use constant or linear laws, then move to more complex expressions.

### Remember that laws are copied, not validated
The generator writes the law strings exactly as you provide them. If the later CSF reader cannot evaluate them, the issue is in the law expression, not in this geometry generator.

### `t_cell` is a naming tag, not a geometric thickness
This is easy to confuse with `tg`, but they are different.

---

## 17. Typical use summary

To generate a variable twisted CSF tower-like section with two internal bar rows:

1. Set `z0` and `z1`
2. Set the `S0` geometry (`s0_*`)
3. Set the `S1` geometry (`s1_*`)
4. Set `twist_deg`
5. Set `N`
6. Choose `singlepolygon=true` or `false`
7. Set row 1 and row 2 bar counts, areas, and placement distances
8. Set `rebar_weight`
9. Optionally set:
   - `bars_row1_law`
   - `bars_row2_law`
   - `s0_law` / `s1_law` for the shell
10. Run the launcher

That is the intended usage pattern of this tool.

## 18. One complete example

```bash
#!/usr/bin/env bash
set -euo pipefail

SCRIPT_DIR="$(cd -- "$(dirname -- "${BASH_SOURCE[0]}")" && pwd)"
PY_SCRIPT="$SCRIPT_DIR/../../src/csf/utils/writegeometry_rio_v2.py"

z0="0.0"
z1="50.0"

s0_x="0.0"
s0_y="0.0"
s0_dx="10.5"
s0_dy="10.5"
s0_R="4.0"
s0_tg="0.25"
s0_t_cell="0.0"

s1_x="0.0"
s1_y="0.0"
s1_dx="5.0"
s1_dy="7.0"
s1_R="1.2"
s1_tg="0.25"
s1_t_cell="0.0"

twist_deg="30.0"
N="128"
singlepolygon="true"

n_bars_row1="60"
n_bars_row2="60"
area_bar_row1="0.001257"
area_bar_row2="0.001257"
dist_row1_outer="0.06"
dist_row2_inner="0.06"
rebar_weight="5.71"

bars_row1_law="np.maximum(0.2, 1.0 - 0.8*(z/L))"
bars_row2_law="np.maximum(0.2, 1.0 - 0.8*(z/L))"
s0_law="1.0 - 0.10*(z/L)"
s1_law="1.0 - 0.10*(z/L)"

out="twist_tower.yaml"
```

This configuration produces:

- a tapered section from `10.5 x 10.5` to `5 x 7`
- rounded corners at both ends
- `30°` twist at the head
- two rows of internal bars
- one law for each bar row
- one paired shell law
- output file `twist_tower.yaml`
